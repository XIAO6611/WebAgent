# 基于多模态大模型与原生插件架构的 Web Agent 系统策划案

**(对应大作业选题：题目7 - 网页多模态Agent：界面理解+自动操作)**

## 1. 项目介绍与相关工作调研

**项目介绍**

本项目旨在构建一个高度拟人化、伴随式的网页多模态 Agent。与传统依赖底层 HTML 源码的自动化脚本不同，本项目以 **Chrome 浏览器插件（Extension）** 为物理基座，结合视觉语言模型（VLM）与大语言模型（LLM），实现“可见即可点”的网页交互。系统能够接收自然语言指令，自主完成跨网页导航、视觉状态识别、复杂表单语义填充以及信息提纯，最终实现真正的“网页端自动驾驶”。

**相关工作调研（结合实际预期对标）**

- **WebVoyager / AppAgent**：这两项研究证明了**纯视觉交互（仅依赖屏幕截图和坐标系）**在网页和移动端的高泛化性。我们的感知模块直接对标此理念，摒弃 DOM 解析，彻底解决网页前端频繁改版导致脚本失效的问题。
    
- **SMART-LLM**：提出了多 Agent 协同。我们的系统吸收了其精髓，但在工程上优化为“单一大模型+动态 Prompt 路由”，将 LLM 划分为规划者（Planner）、质检员（Critic）和总结员（Summarizer），在控制成本的同时实现自我纠错。
    
- **OpenClaw**：开源的 Web Agent 框架。我们参考其动作执行与日志追踪标准，以满足大作业对“可视化操作轨迹与错误分析”的要求。
    

## 2. 现存问题场景、痛点与解决方案

在 Web Agent 的实际落地中，目前业界普遍面临以下四大工程痛点，我们的设计对此逐一给出了根治方案：

1. **痛点一：大厂严苛的反爬与“登录墙”拦截**。传统 Python+Playwright 框架极易泄露自动化指纹，且无法继承用户日常登录状态，在携程、小红书等高防护网站上寸步难行。
    
    - **解决方案**：采用 **Chrome 原生插件架构**。Agent 寄生于用户真实浏览器，共享真实 Cookie 与会话，天然免疫反爬虫机制；遇到未登录状态，可直接调用原生 UI 提醒用户手动扫码。
        
2. **痛点二：非结构化表单映射困难**。各网站表单命名毫无规律（如“毕业院校”与“最高学历”），硬编码正则匹配极易崩溃。
    
    - **解决方案**：引入 **语义映射与本地知识库（RAG）**。将个人简历预先解析为标准 JSON 存储于浏览器本地。VLM 识别表单标签后，由 LLM 进行语义检索，实现跨网站精准填表。
        
3. **痛点三：静态视觉的“交互盲区”**。仅靠一次性截图，无法发现需要鼠标悬停（Hover）才展示的下拉菜单。
    
    - **解决方案**：设计 **“假设-验证”动作机制**。VLM 推断潜在隐藏区域，下达悬停指令，插件重新截图验证子菜单展开后再执行点击。
        
4. **痛点四：自动执行高风险操作的不可控性**。AI 误投递或乱填期望薪资可能引发严重后果。
    
    - **解决方案**：实装 **HITL（人机协同）拦截机制**。在表单最终提交前，内容脚本（Content Script）强行注入核对弹窗，冻结提交流程，等待人类二次确认。
        

## 3. 整体架构设计

系统采用 **“云端强核大脑 + 本地敏捷手眼 + 本地结构记忆”** 的三层架构：

- **感知执行层（手与眼）**：由 Chrome 插件的 Background Worker 和 Content Scripts 构成。负责截取人类真实视口（Viewport）画面、在网页上注入高亮/弹窗 UI，并执行基于绝对坐标的 JS 鼠标事件。
    
- **云端决策层（大脑）**：通过 API 接入强大的多模态大模型（如 Qwen-VL / GPT-4o）。统一负责视觉坐标计算、语义推理、动作规划和文本总结。
    
- **本地数据层（记忆）**：基于浏览器的 IndexedDB 技术，持久化存储用户的多版本简历 JSON 及运行日志。
    

## 4. 各部分功能与连接机制

- **大模型模块（动态分饰四角）**：
    
    - **任务总指挥 (Planner)**：解析用户自然语言，拆解多网页访问步骤，生成目标 URL 列表。
        
    - **视觉导航员 (VLM Actor)**：接收插件传来的 Base64 截图，识别“输入框”、“按钮”，输出具体的动作指令和 `(x, y)` 坐标。
        
    - **状态质检员 (Critic)**：对比期望目标与当前截图。识别是否遭遇“登录墙”、是否“需要人类输入缺失字段”。
        
    - **数据总结员 (Summarizer)**：将抓取到的非结构化文本整理为结构化 Markdown 报告。
        
- **浏览器插件模块（执行与交互）**：
    
    - **通信中枢 (Background.js)**：维护整个任务的状态机，负责调用大模型 API。
        
    - **交互探针 (Content.js)**：注入目标网页，执行点击/输入动作，渲染 HITL 悬浮窗拦截用户提交。
        
- **本地知识库**：
    
    - 存储用户的通用信息（姓名、学历、项目经历 JSON），在填表场景中供 Planner 检索。
        

## 5. 可用场景（泛）与三大核心实现场景

**可用场景（泛指）**：

跨平台比价与竞品分析、多平台新媒体自动分发、CRM 客户数据自动录入、批量发票/报表信息提取填报等一切依赖“看与点”的 Web 办公场景。

**本项目期望实现的主要三个场景**：

- **场景 A**：前往需登录的信息网站（携程/小红书），收集旅游/租房等信息并整理。遇到未登录状态自动阻断并提示用户。
    
- **场景 B**：根据内置知识库，跨平台填写问卷/简历。信息缺失需主动提示用户补充，最终提交须经由用户确认无误（HITL）。
    
- **场景 C**：前往多个不需要登录的开放网页（各大学官网），自动导航并收集考研升学、招聘信息并整理汇总。
    

## 6. 期望运行流程（三大场景细化）

### 场景 A：需登录网站信息收集（如小红书旅游攻略）

- **场景性描述**：用户命令“帮我总结小红书上关于成都春熙路的酒店排雷”。Agent 自动打开小红书，若未登录则提醒用户扫码；登录后自动搜索、向下滚动提取前 5 篇笔记内容，输出总结。
    
- **技术性描述**：
    
    1. 插件 `chrome.tabs.create` 打开小红书主页。
        
    2. VLM 截图分析。Critic 模块识别到画面中央有“扫码登录”特征，阻断流程，在当前网页注入 UI 提示：“检测到登录墙，请手动登录后点击继续”。
        
    3. 用户登录并点击“继续”。VLM 重新截图，确认进入主页，定位搜索框坐标 `(x,y)`。
        
    4. 插件执行 `type("成都春熙路酒店排雷")` 与 `press("Enter")`。
        
    5. VLM 循环识别笔记链接并点击进入，提取文本后退回。Summarizer 最终输出 Markdown。
        

### 场景 B：基于知识库与 HITL 的简历填报

- **场景性描述**：用户进入某未知大厂网申页面。启动 Agent，Agent 自动根据本地简历填好所有教育、实习经历；遇到“期望薪资”时弹窗问用户；最后在点击提交前拦住用户，要求人工核对。
    
- **技术性描述**：
    
    1. VLM 扫描当前表单，提取所有 Input 框的 Label（如“毕业院校”、“期望薪资”）。
        
    2. Planner 将提取到的 Label 与本地 IndexedDB 中的 JSON 简历进行语义匹配。
        
    3. 发现“期望薪资”在 JSON 中不存在。Critic 触发中断，插件注入交互 UI：“缺少期望薪资数据，请手动输入”。
        
    4. 用户输入后，VLM 输出所有待填框的绝对坐标，插件逐一执行 `click()` 与 `type()`。
        
    5. 插件劫持页面上 `type="submit"` 的按钮，将其替换为醒目的 HITL 确认弹窗。用户点击“确认无误”，原生提交事件才被释放执行。
        

### 场景 C：多开放网页信息自动汇编（如考研信息收集）

- **场景性描述**：用户命令“收集清华、北大、上交大的计算机 408 考研招生目录”。Agent 会像幽灵一样自动新建 3 个标签页，分别进入对应官网找寻招生简章，最终生成一份对比表格。
    
- **技术性描述**：
    
    1. Planner 将任务拆解为 3 个子任务（对应 3 所高校）。
        
    2. 插件打开标签页 1（百度搜索“清华 研究生招生网”）。
        
    3. VLM 结合 Hover-Test（悬停试探），在复杂的高校官网导航栏中找到隐藏的“硕士招生”入口并点击。
        
    4. 提取关键文本存入全局状态变量。关闭当前标签页，开启标签页 2 继续执行。
        
    5. 所有子任务完成后，Summarizer 整合全局状态变量，输出 Markdown 对比表格。
        

## 7. 需要的算力资源

- **本地终端**：普通 PC/Mac 即可（内存 $\ge$ 8GB）。插件直接运行在常规 Chrome 浏览器内，无额外环境依赖。
    
- **云端推理（推荐）**：本项目极其轻量化本地算力，核心推理全部交由云端 API。推荐使用具有强大多模态能力的接口（如阿里云 DashScope 的 Qwen-VL-Max，或智谱 GLM-4V）。由于通过代码动态调用，成本极低（单次 API 调用低至几分钱）。
    

## 8. 核心技术和创新点

1. **原生浏览器伴随架构 (Native Browser Extension)**：彻底抛弃 Playwright 自动化驱动，完美继承人类 Cookie 与会话，实现工业级的反反爬虫穿透。
    
2. **基于视口的精准坐标映射 (Viewport-Coordinate Mapping)**：摒弃极易产生 VLM 幻觉的长截图，仅截取人类可视区域，结合页面绝对偏移量计算，实现亚像素级的点击精准度。
    
3. **内生的人机协同机制 (Native HITL)**：在登录检测、数据缺失和高风险提交三个核心节点，首创通过 Content Script DOM 注入进行物理级安全拦截。
    
4. **非结构化表单语义路由 (Semantic Form RAG)**：不依赖页面底层 name 或 ID 标签，纯靠视觉 Label 进行本地 JSON 知识图谱映射，具备极强的跨站泛化能力。
    

## 9. 评测方案 (对标 60+ 次实验验收标准)

为严格满足大作业“至少 60 次独立测试、记录成功率及错误类型分析”的要求，我们将针对上述三大场景进行量化评测：

**评测体系与指标**：

- **任务成功率**：在无需用户干预异常（除正常 HITL 外）的情况下完成闭环的比例。
    
- **错误类型归类**：识别错误（VLM 点错位置）、规划错误（LLM 陷入逻辑死循环）、执行错误（网络延迟或前端弹窗遮挡）。
    
- **HITL 触发准确率**：正确拦截需登录状态和高风险提交动作的成功率。
    

**各场景测试分布（共计 60 次）**：

1. **场景 A 评测（20次）**：
    
    - 在小红书/携程分别各执行 10 次不同参数的信息搜索指令。
        
    - _评测重点_：验证系统识别“登录墙”的准确性，以及面对页面无限滚动（Infinite Scroll）的视觉状态判断能力。
        
2. **场景 B 评测（20次）**：
    
    - 挑选 5 家不同互联网公司（如腾讯、阿里、美团等）的网申表单，各执行 4 次投递。
        
    - _评测重点_：评估本地 JSON 简历在不同命名风格表单下的语义映射准确率；测试 HITL 弹窗能否 100% 在最终提交前拦截动作；测试缺失字段提醒功能。
        
3. **场景 C 评测（20次）**：
    
    - 随机抽取 10 所不同高校的官网，各执行 2 次特定政策检索任务。
        
    - _评测重点_：评估 VLM 面对老旧、复杂官网布局时的导航成功率；检验 Hover-Test 悬停探测机制在不同二级菜单上的有效性。
