**模仿真人浏览过程**
**“类似携程和小红书，收集旅游或者其他信息”**。

在传统的 Python + Playwright 框架下，这几乎是不可能完成的任务（因为例如小红书不登录不让看，Playwright 容易被识别拦截）。但切换到 **Chrome 插件方案** 后，情况发生了质变：

2. **状态完美继承**：用户只需要自己先扫码登录好小红书，唤起插件后，Agent 直接使用已登录的身份去搜帖子、看评论。**彻底解决了登录墙的死局**。
    
3. **完美还原多模态数据流**：队友在架构图里画的“获取网页截图 -> VLM识别 -> 动作生成 -> 执行”，在插件里通过 `chrome.tabs.captureVisibleTab` 和内容脚本注入完美复刻。

### 3、大模型一饰多角与解决方案

#### 多个大模型各司其职
##### 1. 响应速度呈指数级提升（低延迟）
多角色痛点：如果用两个不同的大模型（比如模型 A 做规划，模型 B 做质检），A 规划完后，必须把结果通过网络发给 B，B 再进行推理。中间的网络请求延迟（API 调用通常需要 2-5 秒）叠加起来，会让插件显得非常卡顿。

合并优势：单一模型架构下，所有的逻辑都在同一个大脑里处理。调用次数更少，用户的体感是“Agent 反应极其敏捷，像真人一样丝滑”。对于一个需要高频交互的 Chrome 插件来说，低延迟是及格线。

##### 2. 工程架构极简，防 Bug 神器
多角色痛点：多智能体协同（Multi-Agent）在代码层面的维护简直是灾难。你需要写极其复杂的状态机，处理“模型 A 宕机了怎么办”、“模型 A 和 B 吵起来了陷入死循环怎么办”等无数边界情况。

合并优势：在 Chrome 插件的后台脚本（Background.js）里，你们只需要维护一个核心的 API 通信函数。系统状态流转非常清晰：发指令 -> 收动作 -> 执行 -> 再发指令。开发周期至少缩短一半。

##### 3. 完美贴合大作业的“算力预算”
现实情况：大作业说明中提到推荐算力是“1张 24GB 消费级 GPU 或云平台免费额度”。

合并优势：如果真要在本地部署三个不同的开源模型，24G 显存瞬间就会被撑爆。合并为单一强力模型（如只部署一个 Qwen2.5-VL-7B，或者统一调智谱 GLM-4V 的 API），不仅完全符合算力预算，而且成本极低。


#### **当然，这种“单人分饰多角”的方案也有两个必须正视的工程劣势：**

##### 1. 致命的“自我反思盲区”（既当运动员，又当裁判）

- **劣势说明**：大模型和人类一样，有“自我肯定偏见”。如果它（作为 Planner）规划出了一条错误的点击路线，你立刻让它（作为 Critic）去检查，它大概率会硬着头皮说：“我觉得我刚才选的路线非常完美！”它很难看出自己的逻辑漏洞。
    
- **🎯 我们的解法（上下文阻断法）**： 在代码里，**绝对不要把历史聊天记录发给它进行反思**。当触发质检（Critic）时，你必须像对待一个完全陌生的人一样，发起一次**全新、纯净的 API 请求**。 _只发两样东西：【当前的网页截图】 + 【任务的最终目标】_。 冷酷地问它：“你看这张图，能不能达成目标？差在哪里？” 这样就能完美避开它的自我偏见。
    

##### 2. 提示词（Prompt）工程难度激增

- **劣势说明**：因为你们只有一个模型，如果提示词写得不够严谨，模型可能会产生“角色混淆”。比如它正在做总结（Summarizer），却突然输出了一段点击屏幕的坐标指令（Actor）。
    
- **🎯 我们的解法（强制 JSON 输出）**： 在系统提示词（System Prompt）中，必须下达死命令：“你现在是 XXX 角色。你必须且只能以严格的 JSON 格式输出结果。”只要把输出格式限死，它就不会产生角色混淆的废话。

### 四、其余风险预警与降级策略

#### 风险一：VLM 的“幻觉点击”

现象：VLM 说“登录按钮在 (200, 300)”，但插件点过去发现是个广告。

解决方案（已在方案中提及）： Hover-Test，这是一个非常好的防御性编程。建议强化这一步：点击前让插件截一张 100x100 的局部小图再问一次 VLM：“我要点的按钮是不是在这个红框里？” 这能极大提升演示时的流畅度，避免现场翻车。

#### 风险二：动态内容的等待策略

现象：插件代码跑得比网页渲染快，VLM 截到了一张白屏或者 Loading 骨架屏。

建议：不要用固定的 sleep(2)。利用插件能力监听 document.readyState 或者直接对比前后两张截图的像素差异（结构相似性 SSIM），差异变小了再让 VLM 介入。这部分如果来不及做，用硬编码等待 3 秒作为保底方案，报告里写成“未来可优化的点”。
